## 规范变更流程

规范标准的变更需要谨慎处理：

1. **提案阶段**: 在 Discussion 中发起讨论
2. **评审阶段**: 核心团队评审，收集反馈
3. **试行阶段**: 在 1-2 个 Sprint 内试行
4. **正式发布**: 更新文档，全员通知
5. **强制执行**: 通过 CI 检查确保遵守

## 规范优先级

当不同规范冲突时，按以下优先级处理：

1. **安全规范**（最高优先级）
   - 不得违反任何安全相关规范
   
2. **代码规范**
   - PEP 8、类型注解等基础规范
   
3. **项目特定规范**
   - 本项目定义的额外规范
   
4. **团队约定**（最低优先级）
   - 可根据项目阶段灵活调整

## 违反规范的处理

- **轻微违反**: CI 检查失败，自动提示
- **一般违反**: Code Review 时指出，要求修改
- **严重违反**（如安全规范）: 阻塞合并，必须修复
