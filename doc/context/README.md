# doc/context/ - 会话上下文文档

**文档类型**: 会话上下文  
**适用范围**: 开发会话管理  
**最后更新**: 2026-03-21  
**维护者**: OminiConfig Team

## 用途

本目录用于存储开发会话的上下文信息，确保新会话能够快速了解项目状态和之前的开发进度。

## 核心原则

1. **会话开始**：读取最新的上下文文件，了解项目当前状态
2. **会话结束**：更新上下文文件，记录本次开发的变更和决策
3. **持续积累**：上下文文件是项目知识库的补充，记录会话间的连续性

## 文件结构

```
doc/context/
├── README.md                    # 本文件：目录说明
├── current.md                   # 📌 当前会话上下文（必须保持最新）
├── history/                     # 历史会话上下文归档
│   ├── 2026-03-21_init.md
│   ├── 2026-03-21_refactor.md
│   └── 2026-03-22_feature.md
└── templates/                   # 上下文文档模板
    ├── session_start.md         # 会话开始模板
    └── session_end.md           # 会话结束模板
```

## 文件定义

### 1. current.md（当前上下文）

**状态**: 🔴 必须保持最新  
**更新时机**: 每次开发会话结束时更新  
**用途**: 新会话的第一个读取点

**内容结构**:
```markdown

**最后更新**: YYYY-MM-DD HH:MM  
**会话ID**: [可选，用于追踪]

- 当前版本: v2.0.0
- 主要分支: main
- 最新提交: [commit hash]
```

### 2. history/（历史归档）

每次会话结束后，将 `current.md` 复制到 `history/` 并重命名为 `YYYY-MM-DD_brief.md`。

**命名规范**: `日期_简要描述.md`  
**示例**:
- `2026-03-21_init.md` - 项目初始化
- `2026-03-21_refactor_v2.md` - v2.0 重构完成
- `2026-03-22_add_sse.md` - 添加 SSE 功能

**历史文件用途**:
1. 追踪项目演进历史
2. 回顾重要决策的背景
3. 查找特定功能开发的上下文
4. 审计和复盘

## 工作流程

### 会话开始

```
1. 读取 doc/context/current.md 了解项目状态
   ↓
2. 使用 templates/session_start.md 检查环境
   ↓
3. 明确本次会话目标
   ↓
4. 开始开发
```

### 会话结束

```
1. 完成开发和测试
   ↓
2. 使用 templates/session_end.md 总结会话
   ↓
3. 更新 doc/context/current.md
   ↓
4. 将旧的 current.md 复制到 history/
   ↓
5. 提交所有变更（包括上下文文档）
   ↓
6. 推送至 GitHub
```

## 最佳实践

### DO（应该做）
- ✅ 每次会话结束都更新 `current.md`
- ✅ 保持 `current.md` 简洁，聚焦于当前状态
- ✅ 在 `current.md` 中记录关键决策的理由
- ✅ 定期归档历史上下文文件
- ✅ 使用具体的数据（覆盖率百分比、测试数量等）

### DON'T（不应该做）
- ❌ 让 `current.md` 过时
- ❌ 在 `current.md` 中记录过长的实现细节（应放在代码注释中）
- ❌ 删除历史上下文文件（保留项目演进记录）
- ❌ 提交敏感信息（密码、密钥等）到上下文文档

## 与 AGENTS.md 的关系

- **AGENTS.md**: 定义开发规范和流程（不变的标准）
- **doc/context/current.md**: 记录项目实时状态和进展（持续更新）

两者配合使用：
1. 新会话开始时，先读 AGENTS.md 了解规范
2. 再读 current.md 了解项目状态
3. 开发过程中遵循 AGENTS.md 的规范
4. 会话结束时更新 current.md

## 示例场景

### 场景 1：新成员加入
```
1. 阅读 README.md 了解项目概况
2. 阅读 AGENTS.md 了解开发规范
3. 阅读 doc/context/current.md 了解当前状态
4. 查看 doc/architecture/ 了解架构设计
5. 运行测试确保环境正常
6. 开始开发
```

### 场景 2：隔日继续开发
```
1. 阅读 doc/context/current.md 回顾昨天的工作
2. 查看 "下次会话计划" 了解今天的任务
3. 检查 "遇到的问题" 是否有遗留问题
4. 运行测试确认基准状态
5. 继续开发
```

### 场景 3：长期项目回顾
```
1. 查看 doc/context/history/ 目录
2. 按时间顺序阅读历史文件
3. 了解项目演进和重要决策
4. 识别重复出现的问题
5. 优化开发流程
```

## 维护责任

- **每次开发者**: 更新 `current.md`
- **技术负责人**: 定期清理历史文件（保留最近 20 个或 3 个月）
- **团队**: 确保所有成员遵守上下文更新规范

## 工具建议

可以使用以下工具辅助上下文管理：

1. **Git 钩子**: 在 commit 前检查 current.md 是否已更新
2. **CI 检查**: 确保 PR 中包含上下文更新（如适用）
3. **模板生成器**: 使用脚本自动填充模板中的环境信息
